    <tr><th>Integration Point</th><th>Typical API Surface</th><th>Mandatory?</th><th>Practical Responsibility (What it does)</th><th>Primary Stage</th><th>Notes / Common Variants</th></tr>

  <tbody>
    <tr><td>Runtime/Backend Initialization</td><td>backend_init(config)</td><td>Nice-to-have</td><td>Sets up global runtime state (driver connection, device discovery, logging, allocator defaults) so later calls are faster and consistent.</td><td>Startup</td><td>Some stacks allow lazy init on first compile/run, making this optional.</td></tr>
    <tr><td>Capability & Device Query</td><td>get_device_caps(), enumerate_devices()</td><td>Nice-to-have</td><td>Reports supported precisions, memory limits, concurrency, and feature flags used for partitioning and planning.</td><td>Planning</td><td>Can be hardcoded per target; still valuable for portability.</td></tr>
    <tr><td>Graph/Program Validation (Support Check)</td><td>validate(graph_ir)</td><td>Nice-to-have</td><td>Determines which ops/subgraphs are supported, produces diagnostics, and optionally returns partitioning hints.</td><td>Compilation</td><td>Often implemented in framework side; optional but improves user experience.</td></tr>
    <tr><td>Model/Engine Build (Compile)</td><td>compile(graph_ir, compile_options)</td><td>Mandatory</td><td>Lowers a framework graph/IR into a hardware-executable artifact (engine/binary) including layout decisions and target-specific optimizations.</td><td>Compilation</td><td>May include multi-stage IR lowering internally but must yield an executable result.</td></tr>
    <tr><td>Executable Load/Deserialize (Cache)</td><td>load_executable(blob)</td><td>Nice-to-have</td><td>Loads a precompiled engine from disk/bytes to skip compilation and enable deployment workflows.</td><td>Deployment</td><td>Pairs with save/serialize; frequently used for warm start and CI artifacts.</td></tr>
    <tr><td>Executable Serialize/Save</td><td>serialize_executable(exe) -> bytes</td><td>Nice-to-have</td><td>Exports compiled artifacts for caching, shipping to another machine, or A/B testing builds.</td><td>Deployment</td><td>Optional if the framework provides its own caching mechanism.</td></tr>
    <tr><td>Memory Planning & Buffer Allocation</td><td>allocate(device, size), plan_io(exe)</td><td>Mandatory</td><td>Allocates device buffers for inputs/outputs/temporaries and ensures correct lifetime/aliasing so execution is possible.</td><td>Runtime</td><td>Sometimes split into user-provided allocators + backend internal pools.</td></tr>
    <tr><td>Input/Output Binding</td><td>bind_input(exe, name, buffer), bind_output(...)</td><td>Mandatory</td><td>Associates framework tensors with backend buffers (by pointer/handle) and defines shapes/strides for execution.</td><td>Runtime</td><td>May support name-based, index-based, or signature-based binding.</td></tr>
    <tr><td>Execution / Inference Launch</td><td>run(exe, stream, run_options)</td><td>Mandatory</td><td>Enqueues the compiled executable for execution on the device and returns completion status or an event.</td><td>Runtime</td><td>Must cover end-to-end compute; may support multiple streams/graphs.</td></tr>
    <tr><td>Synchronization & Event Handling</td><td>sync(stream), create_event(), wait_event()</td><td>Mandatory</td><td>Provides correct ordering between host/device and between compute/copy so outputs are valid when consumed.</td><td>Runtime</td><td>Even if simplified, a blocking sync is typically required at minimum.</td></tr>
    <tr><td>Resource Destruction / Finalization</td><td>destroy_executable(exe), backend_finalize()</td><td>Mandatory</td><td>Releases compiled artifacts, buffers, handles, and global runtime resources to avoid leaks.</td><td>Shutdown</td><td>May be reference-counted; but must exist conceptually.</td></tr>
