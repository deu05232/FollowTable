# SharePoint Server 2019 — Hardware Requirements by Deployment Scenario (Example Planning Matrix)

| Deployment scenario (row = one install scenario) | Typical topology / role mix | Minimum RAM | Recommended RAM | CPU (minimum) | System drive free space | Data / log storage guidance | Notes |
| --- | --- | --- | --- | --- | --- | --- | --- |
| Single-server farm (SharePoint + SQL on one box) | All roles + database on same host | 32 GB | 64 GB | 4 cores, 64-bit | 120 GB | Separate volumes for SQL data/logs if possible | For demos, dev, small pilots; not ideal for production scalability |
| Two-server: SharePoint (Web/App) + separate SQL | 1 SharePoint server + 1 SQL server | 16 GB (SP) / 32 GB (SQL) | 32 GB (SP) / 64 GB (SQL) | 4 cores each | 100 GB (SP) / 120 GB (SQL) | SQL: fast storage for tempdb, logs; SP: separate search index disk | Basic production split; easier to scale later |
| Three-tier small farm | 2 Web Front Ends + 1 App server + 1 SQL server | 16 GB per WFE / 24 GB App / 32 GB SQL | 32 GB per WFE / 48 GB App / 64+ GB SQL | 4 cores per server | 100 GB per server | SQL on SSD/RAID; App server needs extra disk for service apps/search | Common production baseline for smaller orgs |
| High-availability farm (N+1) | 2+ WFE, 2+ App, SQL Always On (2 nodes) | 16 GB per WFE / 24 GB per App / 32 GB per SQL node | 32 GB per WFE / 48 GB per App / 64+ GB per SQL node | 4–8 cores per server | 120 GB per server | Dedicated storage for SQL replicas; plan network throughput/latency | Adds redundancy; more network and storage requirements |
| Search-intensive farm | Dedicated Search App server + WFEs + SQL | 24 GB Search / 16 GB per WFE / 32 GB SQL | 64 GB Search / 32 GB per WFE / 64+ GB SQL | 8 cores Search; 4 cores others | 120 GB Search | Large, fast disk for index (e.g., 200–500 GB+) | Search indexing and crawl can dominate CPU/RAM/IO |
| Distributed Cache / social features emphasis | Dedicated Cache server + WFEs/App + SQL | 16 GB Cache (min) + baseline for others | 32 GB Cache + baseline for others | 4–8 cores Cache | 100 GB | Keep cache isolated from heavy IIS workloads | Improves latency; watch memory pressure |
| Development / evaluation VM | Single VM, minimal services | 16 GB | 24–32 GB | 4 cores | 80 GB | Use snapshots; avoid production-like load expectations | For learning/testing only |
| Staging / UAT farm | Mirrors production roles at smaller scale | 16–24 GB per tier | 32–48 GB per tier | 4–8 cores | 100–120 GB | Similar layout to prod; less IO headroom acceptable | Best for validating upgrades and deployments |
| Large content farm (heavy content DBs) | Multiple WFEs/App + scaled-out SQL | 16–32 GB per WFE / 24–64 GB App / 64+ GB SQL | 32–64 GB per WFE / 64+ GB App / 128+ GB SQL | 8+ cores for App/SQL | 120+ GB | SQL storage throughput is critical; separate LUNs/volumes | Driven mainly by DB size, concurrency, and service app mix |
